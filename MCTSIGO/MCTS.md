# 引导树粗搜索 + IGO 细搜索：面向一般组合优化问题的通用框架

> **这不是真正的 MCTS。** 我们借用 MCTS 的**数据结构形式**（树、UCB 评分、Selection/Backprop），但底层模型根本不同：树是静态预建的索引结构，所有离散组合（路径）从一开始就存在，Selection 是在有限路径集中选择下一条要评估的路径，Backprop 是把评估结果登记到路径经过的节点上。不存在"节点价值传播"——所有 cost（包括离散变量的）都是非马尔可夫的，无法在树结构上做加法分解。
>
> 本质是一个**长得像 MCTS 的超大多臂老虎机**：臂 = 预建路径，树 = 路径索引，UCT = 臂选择策略，三种保真度 = 三种精度的臂评估。详见 MCTSforMPC.md §10。

> 从 Unit Commitment 能源系统优化出发，抽象出一套适用于自动驾驶路径拓扑、金融组合选择、芯片布局、物流调度等**任何涉及离散拓扑决策 + 连续参数优化**的组合优化问题的通用求解范式。

---

## 1. 问题的二分结构：离散拓扑 + 连续参数

### 1.1 为什么需要这层抽象

几乎所有有实际意义的组合优化问题，本质上是两层决策的耦合：

| 层 | 决策类型 | 例子 |
|----|---------|------|
| **上层：拓扑/结构** | 离散选择 | 机组启停、路径骨架、资产入选、设备选型 |
| **下层：参数/调度** | 连续优化 | 出力分配、速度曲线、权重配置、时序调度 |

传统 MILP 把这俩塞进同一个线性化模型——代价是必须线性化成本、加 Big-M 约束、场景树削减不确定性。MCTS + IGO 的根本优势在于：**让离散的归 MCTS，连续的归 IGO，各用各的强项，通过双向反馈对齐**。

### 1.2 通用问题形式

给定一个组合优化问题 $\mathcal{P}$：

$$\min_{s \in \mathcal{S}, \; x \in \mathcal{X}(s)} \; f(s, x; \xi) \quad \text{s.t.} \quad g(s, x) \leq 0$$

其中：
- $s \in \mathcal{S}$ — **结构变量**（离散）：启停组合、路径拓扑、资产子集等
- $x \in \mathcal{X}(s)$ — **参数变量**（连续）：给定结构 $s$ 下的资源配置
- $\xi$ — 随机参数（需求、风速、市场价格等）
- $f$ — 可以是任意非凸、非解析的黑箱函数（直接塞进 IGO）
- $g$ — 约束可通过 $\sigma$-饱和嵌套化为连续惩罚（Constran 机制）

## 2. 引导树粗搜：离散拓扑空间的结构化探索

> **树不是逐步构建的，是静态预建的索引结构。** 所有离散组合（路径）从一开始就存在——每个叶节点 = 一条完整路径 = 一个完整的结构方案。Selection 是在有限路径集中选择下一条要评估的路径，Backprop 是把评估结果登记到路径经过的节点上。不存在"节点价值传播"——所有 cost 对离散变量都是非马尔可夫的。本质是一个**长得像 MCTS 的超大多臂老虎机**（臂 = 路径，树 = 索引，UCT = 臂选择策略）。

### 2.1 树的语义：不是逐时步展开，而是逐决策单元展开

关键洞察（来自 UC 经验的泛化）：

> **树深度 = 离散决策单元的数量（结构维度），而非时间步或状态数。每个叶节点 = 一条完整路径。**

| 问题域 | 决策单元 | 树深度 | 每个节点的选择 |
|--------|---------|--------|-------------|
| 能源 UC | 每台机组 | $N_{\text{gen}}$ | 24h 启停区间（开/关区间） |
| 自动驾驶 | 每个路口 | $N_{\text{intersection}}$ | 通行顺序骨架 |
| 投资组合 | 每支资产 | $N_{\text{asset}}$ | 入选/排除 |
| 物流路径 | 每个枢纽 | $N_{\text{hub}}$ | 访问顺序 / 车辆分配 |
| 芯片布局 | 每个模块 | $N_{\text{module}}$ | 区域分配 |

为什么不逐时步展开？
- 逐时步展开 → 树深 $H \times N$，指数爆炸
- 逐决策单元展开 → 树深 $N$，每个叶节点 = 一个完整的结构方案（如 24h 启停区间）
- 结构方案内部有强时序约束（一旦决定了"从何时开到何时关"，中间时步的启停就确定了）
- cost 对离散变量是非马尔可夫的！不能逐时步分解后做加法
- **等价于对搜索空间做了降维：用宏决策取代微决策序列**

### 2.2 分块策略

当决策单元数量大时，直接建一棵深度 $N$ 的树仍太大。解决方案：

**按功能/类型分块（Blockwise Decomposition）**：

```
Block 0: baseload (base units)    — 大容量、低成本、基本一直开着
Block 1: mid_merit (flexible)     — 中等容量、按需启停
Block 2: peakers (marginal)       — 小容量、高峰才开
```

每块独立建索引树，块间通过 Master 协调。分块原则：
1. **块内同质**：同一块的决策单元行为相似，MCTS 内部统计有意义
2. **块间异质**：不同块的角色不同，不需要在同一棵树上互相干扰
3. **块大小受 GPU 约束**：每块深度 ≤ 12，使得路径采样次数可控

### 2.3 UCB1 节点评分（无参数）

树中每个节点的 Selection 本质是**多臂老虎机的臂选择**：$K$ 个分支 = $K$ 条臂，每条臂对应一个路径子集。标准 UCB1：

$$\text{Score}(i) = \begin{cases}
+\infty, & N_i = 0 \\[6pt]
\dfrac{W_i}{N_i} + \sqrt{\dfrac{2 \ln N_{\text{parent}}}{N_i}}, & N_i > 0
\end{cases}$$

其中：
- $W_i$ — 分支 $i$ 的累计胜场（该分支对应路径子集的 cost 越低越"胜"）
- $N_i$ — 分支 $i$ 被评估的总次数
- 标准 UCB1，**无参数**。$N_i=0$ 的臂优先探索

**关键设计**：Score 不依赖任何问题特定的权重参数。引导树只需知道"哪个分支（路径子集）更值得继续评估"，具体好到什么程度由 IGO 回传。

### 2.4 引导树的输出：不是单一解

引导树输出 **top-K 候选结构** + **争议列表**：

- **Top-K 候选**：胜率最高的 K 个完整结构方案，送入 IGO 精评
- **争议列表**：Score 方差大的节点 → 这些决策单元 UCB1 还没信心 → IGO 需要重点诊断

这避免了"引导树选错、IGO 盲从"的单向陷阱——最终裁决是 IGO，引导树只管采样引导。

## 3. IGO 细搜索：连续参数空间的自然梯度优化

### 3.1 IGO 的三重角色

在 MCTS + IGO 框架中，IGO 不只求解——它同时扮演三个角色：

| 角色 | 输入 | 输出 | 调用层级 |
|------|------|------|---------|
| **评估器** | 结构候选 | cost 估计 | 层1 饱和cost快评（μs） |
| **精细求解器** | 结构候选 | 最优连续参数 + cost | 层2 轻量 IGO（3-5s） |
| **诊断器** | 结构 + 参数 | 逐决策单元的瓶颈信号 | 层3 精评 IGO（15-30s） |

### 3.2 三层评估金字塔

```
         ┌─────────────────────┐
         │  层3 精评 IGO        │  ← 3-5 次调用
         │  T=300, B=200        │     全 24h, Chance/DRO, 完整诊断
         │  MC=100, K=3         │
         └─────────┬───────────┘
                   │
         ┌─────────┴───────────┐
         │  层2 轻量 IGO        │  ← 30-50 次调用
         │  T=100, B=30         │     全 24h, 初步诊断
         │  MC=30, K=1          │
         └─────────┬───────────┘
                   │
         ┌─────────┴───────────┐
         │  层1 饱和cost快评    │  ← 数千次调用
         │  同Constran cost     │     微秒级, 无MC, 无IGO迭代
         │  无Chance/DRO        │     纯确定性单点评估
         └─────────────────────┘
```

三层使用**同一个 Constran 构建的饱和嵌套 cost 函数**，只在计算精度上拉开差距——层1 砍掉 MC 采样和 IGO 迭代，层2 加轻量 MC 和少量迭代，层3 全规格。没有建模鸿沟。

**为什么不用 LP 松弛？** LP 松弛要求把非凸成本线性化（SOS2）、把 σ-饱和约束转成 Big-M 不等式——这意味着层1 评估的是一个和层2/3 **拓扑不同**的代理模型。MCTS 在代理模型上选出的 top-K，到了层2 真实模型可能全翻车。用同一个 Constran cost 做三层降精度评估，保证"层1 觉得好的，层2/3 不会觉得差"——这是 MCTS 搜索方向正确的必要条件。

层1 不碰 GPU — MCTS 遍历几千次只需毫秒。层2/3 用 GPU，通过增量式编码 + 块并行求解。

**层1 的 Constran 饱和cost 具体怎么做**（以 UC 为例）：

给定一个结构候选（启停表）$s$，只需计算一次确定性 cost。Constran 的嵌套结构是**递归 $\sigma$-包裹**，不是平坦链。以优先级 P1（最严格）> P2 > P3（最宽松）为例，构建顺序从内到外：

$$\begin{aligned}
\mathcal{L}_0 &= \sigma_{k_{\text{inner}}}\!\left(T(f(x))\right) \quad \text{—— 目标函数先行 log + σ 压缩} \\[4pt]
\mathcal{L}_1 &= \sigma\!\left(T(g_{\text{P3}}) + \mathcal{L}_0\right) \quad \text{—— P3: Soft 模式，违反量直接叠加} \\[4pt]
\mathcal{L}_2 &= \sigma\!\left(\delta_{\text{P2}} \cdot \sigma(\beta_{\text{P2}} \cdot T(g_{\text{P2}})) + \mathcal{L}_1\right) \quad \text{—— P2: Tunable，$\delta$ 控幅度，$\beta$ 控锐度，双重饱和封顶} \\[4pt]
\mathcal{L}_3 &= \sigma\!\left(\mathbb{1}[g_{\text{P1}} > 0] \cdot (T(g_{\text{P1}}) + \delta_{\text{P1}}) + \mathbb{1}[g_{\text{P1}} \leq 0] \cdot \mathcal{L}_2\right) \quad \text{—— P1: Hard，违反则强制 +δ 偏移压倒所有内层}
\end{aligned}$$

其中：
- $T(x) = \text{sign}(x) \cdot \log(1 + |x|)$ — 对数变换，压缩大值范围
- $\sigma(x) = x / \sqrt{1 + x^2}$ — 饱和函数（默认 $k=1$），输出 $\in (-1, 1)$，替代所有 `jnp.where` 硬分支
- **Hard** 是唯一保留 `jnp.where` 的地方：违反时 $T(g) + \delta$ 一定压倒所有内层（$\delta$ 让输出推到 $\sigma$ 饱和区），不违反时原样透传内层。这是一种**结构化的分支**——只在层边界分岔，不在层内部用 `where` 切碎梯度
- **Tunable** 的 $\delta \cdot \sigma(\beta \cdot T(g))$ 在 $g$ 大时自动封顶于 $\delta$，保证单个约束不会无限主导 cost

这就是层2/3 完全相同的 cost 拓扑。层1 只把 Chance/DRO 约束的 MC 采样砍到 1（或直接关掉），确定性约束照常计算。

层1 和层2/3 的唯一区别：

|  | 层1 快评 | 层2 轻量 IGO | 层3 精评 IGO |
|--|---------|-------------|------------|
| 连续参数 $x$ | 固定值（如全 0 或 warm start） | IGO 优化 T=100 步 | IGO 优化 T=300 步 |
| MC 采样 | 无（点估计） | 30 个 | 100 个 |
| 不确定性 | 忽略 | Chance 约束 | Chance + DRO |
| 耗时 | μs | 3-5s | 15-30s |
| Cost 函数 | **同一个 Constran 函数** | 同一个 | 同一个 |

关键保障：层1 的 cost 对层2/3 的 cost 有**保序性**——如果 $s_A$ 在层1 比 $s_B$ 好很多，层2/3 极大概率也是 $s_A$ 更好。这不是数学定理，而是在 Constran 连续松弛下的工程经验：σ-饱和嵌套让 cost 曲面对结构变化响应光滑，不存在 LP 松弛那种"松弛最优解在真实模型下完全不可行"的崩塌。

### 3.3 块并行 IGO 求解器（Blockwise MGIGO）

对于 24h 联合优化，连续变量维度可能很大（50台机 × 24h = 1200 维）。解决方案：

**分块求解，联合采样**：

```
决策量 = 24 × (N_gen + N_batt + N_wind + N_solar + N_h2)

分块：每块 D ≤ 10
  Block 0:  hour 0, baseload gens (D=7)
  Block 1:  hour 0, mid+peak gens (D=7)
  Block 2:  hour 0, 储能+弃电 (D=6)
  Block 3:  hour 1, baseload gens (D=7)
  ...
  Block 47: hour 23, 储能+弃电 (D=6)

总块数 M = 48, 每块独立 GMM (K=3, D≤10)
→ 48 × 3 = 144 个低维高斯分量
→ VRAM ~3-5 GB (RTX 5070 充裕)
```

每块维护独立的 GMM $(\mu_j, S_j, \pi_j)$，$j=1,\dots,M$。采样时从各块独立抽取 → 拼接成全维解。更新时每块独立做自然梯度步。**关键是块间通过同一个 cost 函数的反馈对齐**——不需要显式的块间通信。

### 3.4 诊断信号：IGO 不只返回 cost

这是 MCTS + IGO 双向反馈的核心。IGO 返回**逐决策单元的瓶颈诊断**：

| 诊断信号 | 检测方法 | 含义 | 传给引导树的动作 |
|---------|---------|------|-----------------|
| 容量到顶 | $\text{mean}_t(x_{i,t}) > 0.95 \cdot x_i^{\max}$ | 该单元开少了 | 对应 ON 节点败率 ↑ |
| 长期低负荷 | $\text{mean}_t(x_{i,t}) < 1.1 \cdot x_i^{\min}$ | 该单元开多了 | 对应 ON 节点败率 ↑ |
| 储能触底/顶 | $\text{freq}(\text{SoC} \in \partial)$ | 容量配错 | 调整储能块参数 |
| 过度弃电 | $\text{mean}_t(\text{curtail}_t) > \tau$ | 消纳不足 | 更新 curtail 偏好 |
| 约束瓶颈 | $\text{mean}_t(\text{viol}_t) > 0$ | 某类约束太紧 | 调整约束优先级 |

诊断信号的设计原则：
1. **每单元独立** — "Gen 5 卡在 p_max" 只影响 Gen 5 的 MCTS 节点，不污染其他
2. **连续而非二值** — 信号强度反映问题严重程度
3. **可回溯** — 信号指出"哪里出问题"，不仅"有没有问题"

### 3.5 GMM 保持 K=3：利用 + 探索

IGO 的 GMM 保持 $K=3$ 分量：
- **分量 0**：贴近 MCTS 建议（利用现有结构）
- **分量 1, 2**：自由探索不同的连续参数配置

如果自由分量发现更好的 dispatch 模式（如不同的储能充放策略），不影响启停结构——但影响 cost → 影响 MCTS 的胜负回传。

## 4. 双向反馈闭环：为什么单向不行

```
┌──────────────────────────────────────────────┐
│          引导树（粗搜，离散路径采样）             │
│  静态索引树 + UCB1 臂选择 | 所有路径预建          │
│  输出: top-K 候选结构 + "争议决策单元" 列表      │
└──────────────────┬───────────────────────────┘
                   │ K 个候选结构
                   ▼
┌──────────────────────────────────────────────┐
│              IGO（精搜 + 诊断，连续）            │
│  对每个候选，联合优化所有连续参数                │
│                                              │
│  诊断信号（非仅 total cost）:                   │
│   容量到顶 → 开少了                            │
│   长期低负荷 → 开多了                          │
│   储能边界 → 容量配错                          │
│   过度弃电 → 消纳不足                          │
│                                              │
│  输出: cost + 诊断向量（逐单元）                │
└──────────────────┬───────────────────────────┘
                   │ cost + 诊断
                   ▼
┌──────────────────────────────────────────────┐
│           引导树更新（路径统计量回传）            │
│  诊断 → 树统计量更新:                           │
│   "容量到顶" → ON 分支胜率 ↓（应多开）           │
│   "长期低负荷" → ON 节点败率 ↑（应关）           │
│   cost 显著优于预期 → 沿路径胜率 ↑               │
│                                              │
│  引导树重新采样 top-K → 下一轮                   │
└──────────────────────────────────────────────┘
```

三个关键保障：

1. **引导树输出争议列表，不只一个解**。如果引导树把错误的唯一解传给 IGO，IGO 只能在那个解的领域优化——无法推翻。Top-K + 争议列表让 IGO 有选择权。最终裁决是 IGO，不是引导树。

2. **IGO 诊断是每单元独立的**。"Gen 5 出力卡 p_max" 只影响 Gen 5 对应分支的统计量，不污染 Gen 3。这使得搜索可以"纠错不推翻"——只修正有问题的那一小部分，而非重来。

3. **收敛判据**：当引导树的 top-K 候选连续两轮 cost 变化 < 1%，且争议列表为空（所有单元 Score 方差都小），停止。

## 5. 各问题域的映射

### 5.1 自动驾驶：路径拓扑 + 速度曲线

| 组件 | 映射 |
|------|------|
| **结构 $s$** | 路口通行顺序骨架 / 行为树（左转、直行、变道） |
| **参数 $x$** | 速度曲线、加速度剖面、跟车距离 |
| **MCTS 深度** | $N_{\text{intersection}}$（非 $H \times$ 状态数） |
| **IGO 分块** | 各路段独立的速度曲线块 |
| **诊断信号** | 加速度饱和（路径太保守）、刹车频繁（路径太激进）、安全距离触底 |
| **不确定性 $\xi$** | 他车行为、行人穿越、感知噪声 |

### 5.2 金融组合优化：资产入选 + 权重配置

| 组件 | 映射 |
|------|------|
| **结构 $s$** | 资产入选子集 / 行业板块选择 |
| **参数 $x$** | 组合权重、再平衡频率 |
| **MCTS 深度** | $N_{\text{asset}}$（逐资产决定入选/排除） |
| **IGO 分块** | 入选资产按行业分块，每块优化权重 |
| **诊断信号** | 权重触上下限（入选太多/太少）、VaR/CVaR 瓶颈、换手率过高 |
| **不确定性 $\xi$** | 收益率分布、相关性变化、尾部事件 |

### 5.3 芯片布局：模块分配 + 布线参数

| 组件 | 映射 |
|------|------|
| **结构 $s$** | 模块→区域分配 / 拓扑连接图 |
| **参数 $x$** | 模块坐标、绕线长度、功耗分配 |
| **MCTS 深度** | $N_{\text{module}}$（逐模块决定区域） |
| **IGO 分块** | 各区域模块独立优化坐标 |
| **诊断信号** | 面积到顶、时序违规、热点聚集、拥塞触限 |

### 5.4 物流调度：车辆路径骨架 + 装载/时间窗

| 组件 | 映射 |
|------|------|
| **结构 $s$** | 车辆→客户分配 + 访问顺序骨架（TSP 子问题） |
| **参数 $x$** | 装载量、到达时间、速度 |
| **MCTS 深度** | $N_{\text{vehicle}} + N_{\text{hub}}$ |
| **IGO 分块** | 每车的路径用 Plackett-Luce IGO 或 Categorical IGO |
| **诊断信号** | 时间窗违规、容量到顶、车辆闲置 |

## 6. 实时MPC vs 大规模优化：同一父框架的子类分化

§2–§4 描述的是 MCTS + IGO 的**父框架**——离散/连续二分、UCB1 树搜索、三层评估金字塔、诊断回传闭环。这个父框架对两类极端场景都成立，但**层配置、反馈轮次、时间分配、样本复用策略**会分叉成两个子类。

### 6.1 两类场景的根本差异

| 维度 | 实时 MPC（自动驾驶） | 大规模优化（能源 UC 50+机组） |
|------|---------------------|---------------------------|
| **单次求解时间预算** | 1–50 ms | 1–30 s |
| **重规划频率** | 每时步（10–100 Hz） | 一次性 / 每15分钟 |
| **决策单元数 $N$** | 3–15（几个路口/行为） | 30–100（大量机组/资产） |
| **树深度** | $N \leq 15$，浅 | $N$ 可达 50+，需分块 |
| **不确定性** | 感知噪声、他车行为 | 需求/风光分布 |
| **前一解与当前解的相关性** | 极高（时步间差 100ms） | 低至中（15min 后需求变了） |
| **硬件** | 车载芯片（Orin/Thor） | GPU 服务器（RTX 5070+） |

### 6.2 共享的父框架基因

无论哪个子类，以下不变：

```
┌─────────────────────────────────────────┐
│  ① 离散/连续二分                          │
│     MCTS 管结构 s，IGO 管参数 x            │
├─────────────────────────────────────────┤
│  ② Constran 饱和嵌套 cost                 │
│     同一 cost 函数，不同精度配置             │
├─────────────────────────────────────────┤
│  ③ 宏决策降维                             │
│     树深 = 决策单元数，不是时步×单元         │
├─────────────────────────────────────────┤
│  ④ 分块求解                               │
│     块内 GMM (K=3, D≤10)，块间 cost 对齐   │
├─────────────────────────────────────────┤
│  ⑤ 双向反馈闭环                           │
│     MCTS 输出候选 → IGO 评估+诊断 → 回传    │
├─────────────────────────────────────────┤
│  ⑥ IGO 自然梯度更新                       │
│     blockwise_mgigo 核心步，与问题无关      │
└─────────────────────────────────────────┘
```

### 6.3 子类 A：实时 MPC 组合优化

**场景**：自动驾驶路径拓扑、机器人任务序列、实时交易

**核心特征**：离散节点少（$N \leq 15$），但**非马尔可夫代价（启动费、最小启停时间、储能 SoC 继承、爬坡约束）使得当前时步的拓扑决策与未来时步深度耦合**。MCTS.md §6.3 的原始方案——"MCTS 决定 $s_t$，串行评估各候选"——忽略了这个耦合，导致候选评估本身建立在错误的未来拓扑假设上。

**正确方案**：时域联合 MCTS + 多模态 IGO。详见 **[MCTSforMPC.md](MCTSforMPC.md)**。

要点速览：

| 维度 | 错误做法 | 正确做法 |
|------|---------|---------|
| MCTS 搜索空间 | 当前时步 $s_t$ | 时域 $[t, t+H]$ 的拓扑序列 $\mathbf{s}_{t:t+H}$ |
| IGO 评估 | 串行 K 次，每次固定 $s_t$ | 单次多模态 IGO（K 分量），全时域联合优化 |
| 候选计算 | K 次独立 IGO，重复计算 | GMM 分量间共享梯度，一次运行覆盖多个模态 |
| 非马尔可夫项 | IGO 隐式处理 | 时域前向 rollout 显式求和 |
| Warm Start | GMM + 树统计量 | + 时域滑窗裁剪 + 跨时步诊断记忆 |
| 树结构 | 深度 $N$ | 深度 = 时域内拓扑变更事件数（$\ll H \times N$） |

**为什么这样设计**：
1. **$s_t$ 无法独立评估** — 评估 Gen 3 开机 vs 关机需要知道后续时步能不能关/开，这依赖尚未做出的未来离散决策
2. **串行 IGO 浪费计算** — 两个拓扑候选的连续参数大量重叠（Gen 1,2,4,5 的出力分配类似），被独立跑了两遍
3. **多模态 IGO 天然适合** — $K>1$ 分量在同一优化中探索不同拓扑模式，分量间通过 Mixture 分母共享梯度信息，比串行收敛更快

参数配置和工程细节见 [MCTSforMPC.md](MCTSforMPC.md) §2–§4。

### 6.4 子类 B：大规模离线/半在线优化

**场景**：能源 UC（50+机组）、金融组合（100+资产）、芯片布局

**核心特征**：离散节点多（$N \geq 30$）→ MCTS 是瓶颈。与实时 MPC 刚好相反——**瓶颈在离散搜索而非连续优化**。连续参数维度虽大但可块并行，IGO 单次评估的成本远低于 MCTS 遍历几千个离散组合的成本。所以需要三层金字塔把昂贵的 IGO 精评集中在少数候选上。

**分化策略**：

**1. 三层都用上**。层1 过滤明显差的候选，层2 批量评估中等候选，层3 只对最终入围的 2–3 个做精评 + 完整诊断。这就是 §3.2 的标准配置。

**2. 分块是刚需**。50 台机组不能在一棵树上——深度 50 的 MCTS 遍历一次都要几百步。分 3–5 块，每块深度 6–12，块间 Master 协调。详见 §2.2。

**3. MCTS–IGO 多轮迭代**。有时间跑 2–3 轮完整闭环：
- Round 1：MCTS 粗选 → IGO 评估 → 诊断回传
- Round 2：MCTS 修正争议单元 → IGO 再评 → 诊断更新
- Round 3（可选）：最终精评 + 完整诊断报告

**4. 不确定性全规格**。Chance + DRO 都用上。100 个 MC 样本 + 多批次分位数估计，DRO 的模糊集包含多个候选分布。这是离线优化的优势——等得起。

**5. 候选流水线并行**。K=5 个候选的层2 评估可以串行（单 GPU VRAM 友好），也可以多进程并行（多 GPU 场景）。详见 §7.4。

**大规模优化的参数速查**：

| 参数 | 30 台机 (2 blocks) | 50 台机 (3 blocks) |
|------|-------------------|-------------------|
| 块数 × 块深 | 2×8 | 3×6 |
| 层1 评估 | ~8000 次 | ~12000 次 |
| 层2 IGO | T=100, B=30, MC=30, K=3 | T=100, B=30, MC=30, K=3 |
| 层3 IGO | T=300, B=200, MC=100, K=3 | T=300, B=200, MC=100, K=3 |
| MCTS–IGO 轮数 | 2 | 2–3 |
| Warm Start | GMM 块级复用（只 reset 受影响的块） |
| 总耗时 | ~10s | ~20s |
| 硬件 | RTX 5070, VRAM 3–5 GB |

### 6.5 两个子类的代码复用

关键点：**父框架代码 100% 复用，子类只在配置层分化**。不需要两套求解器。

```python
from mcts_igo import MCTSIGOSolver, RealTimeConfig, LargeScaleConfig

# 自动驾驶 — 实时配置（离散节点少 → MCTS便宜, IGO必须收敛好）
solver_rt = MCTSIGOSolver(
    RealTimeConfig(
        n_blocks=1,
        layer2_enabled=False,              # 不需要中间过滤层（离散空间小）
        layer3_T=300, layer3_B=80,         # IGO 必须充分收敛 注意不管怎么样 B都必须是4倍决策量 B0大概是B的一半略小
        layer3_MC=20, K=3,
        mcts_igo_rounds=1,                 # 单轮，诊断跨时步回传
        warm_start_gamma=0.9,              # 树 + GMM 双重复用
    )
)

# 能源 UC — 大规模配置（离散节点多 → MCTS瓶颈, 需三层过滤）
solver_ls = MCTSIGOSolver(
    LargeScaleConfig(
        n_blocks=3, block_depth=[6, 6, 6],
        layer2_T=100, layer2_B=30, layer2_MC=30,  # 轻量过滤
        layer3_T=300, layer3_B=200, layer3_MC=100, # 最终精评
        K_layer2=30, K_layer3=3,
        mcts_igo_rounds=2,
        warm_start_gamma=0.0,              # 一次性求解，无跨时步继承
    )
)

# 注：大规模问题，例如能源UC 最后被认为可精修的叶 可以按需不停地调用求解器 直到认为优化结果不错或者再也下不去为止。
# 同一个 Solver 类，同一个 cost_fn 接口
result_rt = solver_rt.solve(structure_gen, cost_fn, problem_rt)
result_ls = solver_ls.solve(structure_gen, cost_fn, problem_ls)
```

唯一真正需要不同代码的插件：
- **实时 MPC**：Warm Start 逻辑（树统计量衰减继承 + GMM 参数跨时步传递 + 诊断跨时步回传）
- **大规模离线**：分块 Master 协调（跨块约束校验 + 容量不足时通知相关块加开）

但这些都是父框架的插件，不影响求解器内核。

## 7. 工程实现：并行计算与样本复用

### 7.1 三层并行粒度

```
粒度 1 — 跨候选并行（MCTS 输出 K 个候选 → K 路 IGO 同时跑）
   ↓
粒度 2 — 块内并行（每个 IGO 内部：M 个块独立采样和更新，vmap 并行）
   ↓
粒度 3 — 样本并行（每块内部：B 个 Monte Carlo 样本，jax.vmap 批量化）
```

**实现策略**：

| 粒度 | 并行方式 | 开销 | 限制 |
|------|---------|------|------|
| 跨候选 | 多进程 / JAX `pmap` | 各进程独立 VRAM | 受 CPU 核数和 GPU 显存限制 |
| 跨块 | `jax.vmap` 块维 | 零额外开销（编译融合） | 块数受 trace 长度限制 |
| 跨样本 | `jax.vmap` 样本维 | 零额外开销 | B 受 VRAM 限制（MC 的 vmap 展开） |

推荐的默认配置（50 决策单元）：
- **不要**跨候选并行——层1 已经筛出 K 个候选，串行跑层2 IGO（5s × K ≈ 15-25s），不需要多进程的复杂度
- **块并行**：用 `vmap` 天然并行，48 blocks 一次 `lax.scan` 搞定
- **样本并行**：`vmap` 批量化采样的精英筛选

### 7.2 样本复用：跨迭代的 old-sample reuse

IGO 的每一轮迭代（step $t$）采样 $B_n$ 个新样本 + 复用上一轮的 $B_o$ 个旧样本。参考 `MPC_ReuseOldSamples.py` 的双分母机制：

$$\tilde{a}_{k}^{(t)}(z_b) = \frac{p_k^{(t)}(z_b)}{\begin{cases} \sum_{l} \pi_l^{(t)} p_l^{(t)}(z_b), & b < B_n \text{ (来自当前采样)} \\ \sum_{l} \pi_l^{(t-1)} p_l^{(t-1)}(z_b), & b \geq B_n \text{ (来自上一轮旧样本)} \end{cases}}$$

**关键**：分母根据样本的"出生轮次"选择对应的分布——新样本用当前分布，旧样本用生成它的旧分布。这保证了重要性采样权重无偏。

**收益**：
- 样本利用率提升 $\frac{B_n + B_o}{B_n}$（通常 2-3 倍）
- 等效于在不增加采样开销的情况下增大了有效样本量
- 旧样本的 cost 评估已经算过（对给定结构 $s$），复用零额外计算

**适用条件**：
- 结构 $s$ 未变（旧样本对应当前的结构评估仍然有效）
- 当 MCTS 反馈修正结构时，旧样本作废，重新采样

### 7.3 计算框架复用：JAX 单进程多路复用

不同于典型的多进程求解器，IGO 的所有计算在 JAX 的 JIT 编译框架内完成：

```
单个 JAX 进程
├── JIT 编译缓存 (~1-2 GB VRAM, 所有求解器共享)
│   ├── blockwise_mgigo (复用于所有结构候选)
│   ├── Constran cost_fn (一次编译, 所有候选复用)
│   └── stochastic 评估 (MC vmap)
│
├── 数据流
│   MCTS 输出结构 s → IGO cost_fn(s, x) → 自然梯度更新 → 诊断信号
│   不同 s 复用同一个 JIT 编译的 cost_fn，只改变输入参数 x
│
└── VRAM 管理
    XLA_PYTHON_CLIENT_ALLOCATOR=platform  # 按需分配，不预占
    层3 MC vmap: 0.5-1 GB (逐个候选串行，用后释放)
    层2 轻量: ~500 MB/次
    总 VRAM 预算: 3-5 GB (RTX 5070 充裕)
```

**框架复用的4个层面**：

1. **Cost Function 复用**：Constran 构建的 cost_fn 一次 JIT 编译，所有结构候选共享。改变结构 $s$ 只是改变 cost_fn 的上下文参数，不需要重新编译。

2. **求解器内核复用**：`blockwise_mgigo` 的核心更新步（`_update_block_k`）是纯 JAX 函数，与问题无关。换一个问题只需换 cost_fn——求解器代码零改动。

3. **GMM 参数复用**：当 MCTS 反馈只修正了某几个决策单元时，上一轮 IGO 的 GMM 参数可以作为 Warm Start——只 reset 受影响的块。

4. **Monte Carlo 样本复用**：在 Chance/DRO 评估中，100 个 MC 样本的 `vmap` 展开可以在多次 IGO 迭代中复用（只要随机种子固定，样本可重现），节省重采样开销。

### 7.4 候选级的流水线

如果 K 个候选串行评估太慢（比如层3 精评每个 30s，5 个候选 = 150s），可以考虑：

```
方案 A — GPU 分时复用（推荐）:
  K 个候选串行跑，每个用 GPU 100%。
  优点：VRAM 占用小，无竞争
  适用：K ≤ 5, GPU 充裕

方案 B — 多流并行 (JAX multi-stream):
  在单 GPU 上用 CUDA streams 并行跑多个 IGO 实例
  限制：JAX 原生不支持多流，需手动管理 CUDA stream

方案 C — 多 GPU (pmap):
  每个 GPU 跑一个候选的 IGO
  开销：跨 GPU 通信（仅在收集结果时）
  适用：K ≥ 4, 多 GPU 可用

方案 D — 早停 + 渐进筛选:
  K 个候选先用层2 轻量 IGO (5s) 评估 → 筛到 2-3 个 → 层3 精评 (30s × 2-3)
  总时间: K × 5s + 2 × 30s ≈ 85s (K=5)
```

### 7.5 跨问题域的求解器共享

整个求解器堆栈可以做成**问题无关的库**：

```python
# 问题定义（用户只需提供）
def my_structure_generator(problem) -> list[Block]: ...
def my_cost_fn(x, structure, context) -> float: ...  # JAX 兼容

# 求解器（通用）
from mcts_igo import MCTSPlanner, IGOFineSolver, MasterCoordinator

planner = MCTSPlanner(
    n_blocks=len(blocks),
    block_depth=[len(b) for b in blocks],
    C=0.3,            # UCB1 探索系数
    K=3,              # top-K 候选数
)

solver = IGOFineSolver(
    M=48, D_max=10, K=3,   # 分块参数
    T2=100, B2=30, MC2=30,  # 层2 参数
    T3=300, B3=200, MC3=100,# 层3 参数
    reuse_ratio=0.5,         # 样本复用比例
)

coordinator = MasterCoordinator(planner, solver)
result = coordinator.solve(my_structure_generator, my_cost_fn, problem)
```

## 8. 与传统方法的对比

| 维度 | MILP (Gurobi) | 纯 IGO | MCTS + IGO |
|------|--------------|--------|-----------|
| **离散搜索** | B&B 枚举 | 连续编码隐式 | UCB1 树搜索 + 分块 |
| **连续优化** | 必须线性化 | 自然梯度 (真实模型) | 自然梯度 (真实模型) |
| **不确定性** | 场景树 (5-10 场景) | Chance/DRO 天然 | Chance/DRO 天然 |
| **成本函数** | 分段线性 (SOS2) | 精确非凸 | 精确非凸 |
| **全局性** | 证明最优 (简化模型) | 局部最优 | 概率近似全局 (真实模型) |
| **规模上限** | ~100 台机 | ~10 台机 | ~50-100 台机 |
| **单次求解** | 30-120s | 2.5s | ~20s |
| **建模负担** | 高（线性化、Big-M） | 低（黑箱） | 低（黑箱 + 结构定义） |

**MCTS + IGO 的适用条件**：
- 问题有明确的离散/连续二分结构 ✓
- 连续部分对固定结构可高效求解（IGO） ✓
- 离散部分的决策单元数 ≤ 100（树深度可控） ✓
- 成本函数可黑箱评估（不需要解析梯度） ✓
- 不确定性需要 Chance/DRO 处理而非简单场景树 ✓

## 9. 工程最佳实践总结

### 9.1 结构设计

1. **先分块，后建树**：按功能/类型把决策单元分块，每块独立 MCTS。块大小 ≤ 12。
2. **宏决策降维**：树的每个节点 = 一个决策单元的完整结构方案（如启停区间），不是单个时步的选择。
3. **争议驱动再搜索**：MCTS 输出争议列表 → IGO 重点诊断争议单元 → MCTS 重新探索。形成"探索-诊断-修正"闭环。

### 9.2 求解器配置

4. **三层评估，同一 cost 不同精度**：层1 饱和cost快评（数千次，μs，砍掉 MC 和迭代）→ 层2 轻量 IGO（30-50 次，5s）→ 层3 精评（3-5 次，30s）。三层用同一个 Constran cost 函数，无建模鸿沟。85% 的 IGO 调用在层2。
5. **成本换精度**：层2 用 T=100 B=30 MC=30，层3 用 T=300 B=200 MC=100。不要在层2 用层3 的参数。
6. **诊断而非仅打分**：IGO 必须返回逐单元诊断信号，而非仅 total cost。诊断是双向反馈的燃料。

### 9.3 工程效率

7. **单进程 JAX，不搞多进程**：所有并行在 `vmap` / `lax.scan` 内完成，零 IPC 开销。
8. **样本复用**：$B_n + B_o$ 采样，旧样本重要性加权复用，利用率提升 2-3 倍。
9. **Warm Start**：MCTS 反馈只修正部分单元时，未受影响的 GMM 块保留参数不动。
10. **VRAM 预算**：`XLA_PYTHON_CLIENT_ALLOCATOR=platform` + 串行候选评估，VRAM 3-5 GB。

## 10. 替代/补充方案

当 MCTS + IGO 不太适合时（如结构空间太小或太大），考虑：

| 方案 | 适用场景 | 核心思路 |
|------|---------|---------|
| **Beam Search** | 结构空间小、启发式强 | 每层保留 K 个最优，确定性强 |
| **Simulated Annealing** | 结构空间连续可变 | 扰动结构 + IGO 评估邻域 |
| **Genetic Algorithm** | 结构可交叉/变异 | 启停方案交叉 + IGO 评估 fitness |
| **Categorical IGO** | 纯离散问题 | 用 `Pure_discrete.py` 直接优化离散变量 |
| **Plackett-Luce IGO** | 排列问题 (TSP 等) | 用 `TSP.py` 优化序列 |
